## 1.1 CPU/GPU 运行模式详解

### device_mode 参数说明

`device_mode` 参数允许用户选择模型的运行模式，提供了灵活的硬件适配方案。

### GPU 模式（推荐）

**适用场景**：
- 有可用 GPU 显存
- 需要快速推理
- 处理大型模型或复杂任务

**特点**：
- 推理速度快，适合实时应用
- 支持更大的模型和更长的上下文
- 自动进行显存估算和优化
- 根据显存大小智能调整参数

**参数设置**：
- `n_gpu_layers`：控制加载到 GPU 的模型层数，-1 表示全部加载
- `vram_limit`：显存限制（GB），-1 表示无限制

**推荐配置**：
- 24GB+ 显存：n_gpu_layers = -1, vram_limit = 24
- 16GB 显存：n_gpu_layers = -1, vram_limit = 16
- 12GB 显存：n_gpu_layers = -1, vram_limit = 12
- 8GB 显存：n_gpu_layers = 30, vram_limit = 8

**性能表现**：
- 推理速度：快（取决于 GPU 性能）
- 显存占用：高（根据模型大小）
- 适用任务：所有类型，特别是实时应用

### CPU 模式

**适用场景**：
- 无 GPU 或 GPU 显存不足
- 需要使用 CPU 进行推理
- 低性能硬件（<8GB 显存）
- 需要最大兼容性

**特点**：
- 不依赖 GPU 显存
- 自动忽略 GPU 相关参数
- 推理速度较慢，但兼容性好
- 适用于所有硬件配置

**参数设置**：
- CPU 模式下，n_gpu_layers 和 vram_limit 参数会被自动忽略
- 无需手动调整这些参数
- 系统会自动设置 n_gpu_layers = 0, vram_limit = -1

**推荐配置**：
- 适用于所有硬件配置
- 适合小型模型和简单任务
- 显存不足时的备选方案

**性能表现**：
- 推理速度：慢（取决于 CPU 性能）
- 显存占用：低（仅使用系统内存）
- 适用任务：简单任务、离线处理

### 智能默认值

插件会根据硬件性能自动选择合适的运行模式：

- **高性能硬件**（8GB+ 显存）：默认使用 GPU 模式
- **低性能硬件**（<8GB 显存）：默认使用 CPU 模式
- **无 GPU 检测**：默认使用 CPU 模式

### 使用建议

1. **优先使用 GPU 模式**：
   - 如果 GPU 显存充足，优先选择 GPU 模式以获得更好的性能
   - GPU 模式推理速度通常比 CPU 模式快 5-10 倍

2. **显存不足时切换 CPU**：
   - 如果遇到显存不足错误（OOM），可以尝试切换到 CPU 模式
   - CPU 模式虽然慢，但不会受显存限制

3. **灵活切换**：
   - 可以根据任务需求随时切换运行模式
   - 不同任务可以使用不同的运行模式

4. **监控性能**：
   - 使用 GPU 模式时，注意监控显存使用情况
   - 使用任务管理器或 GPU 监控工具查看显存占用

5. **参数优化**：
   - GPU 模式：可以调整 n_gpu_layers 和 vram_limit 来优化性能
   - CPU 模式：无需调整 GPU 相关参数，专注于其他推理参数

### 常见问题

**Q: 什么时候应该使用 CPU 模式？**
A: 当遇到以下情况时建议使用 CPU 模式：
- 没有可用的 GPU
- GPU 显存不足以加载模型
- 需要处理多个大型模型，显存不足
- 需要最大兼容性

**Q: GPU 模式和 CPU 模式的性能差异有多大？**
A: 通常情况下，GPU 模式的推理速度比 CPU 模式快 5-10 倍，具体取决于硬件配置和模型大小。

**Q: CPU 模式下 n_gpu_layers 参数还有作用吗？**
A: 没有。CPU 模式会自动忽略 n_gpu_layers 和 vram_limit 参数，强制使用纯 CPU 运行。

**Q: 如何知道当前使用的是 GPU 还是 CPU 模式？**
A: 查看控制台日志输出，会显示当前使用的设备模式：
- GPU 模式：`【设备模式】使用 GPU 模式（n_gpu_layers=X, vram_limit=XGB）`
- CPU 模式：`【设备模式】使用 CPU 模式（忽略 n_gpu_layers 和 vram_limit 参数）`

**Q: 可以在运行时动态切换 GPU 和 CPU 模式吗？**
A: 可以。每次重新加载模型时，可以切换 device_mode 参数。但需要注意的是，切换模式会重新加载模型，可能需要一些时间。

- **Qwen - Image Layered**：专为Qwen-Image-Layered模型设计的Prompt工程师，擅长创建详细的分层提示，利用模型处理复杂构图和多个元素的能力

## 3. Llama-cpp 参数设置节点

### 功能
设置 LLM 推理的详细参数，控制生成文本的质量和风格。

### 参数说明

参数名：最大token数 
 调节范围：0-4096 
 作用说明：最大生成 token 数，控制生成文本的最大长度，直接影响输出内容的完整性 
 推荐设置：低性能设备：512-1024，高性能设备：1024-2048 
 实用建议：
  - 简短回答：256-512
  - 详细描述：768-1024
  - 长文本生成：1024-2048

参数名：采样候选数 
 调节范围：0-1000 
 作用说明：采样候选词数量，控制每次生成时考虑的词汇数量，越小越聚焦，越大越多样 
 推荐设置：低性能设备：20，高性能设备：30 
 影响分析：
  - 低 top_k（<20）：生成更确定，适合事实性任务
  - 高 top_k（>50）：生成更多样，适合创造性任务

参数名：核心采样阈值 
 调节范围：0.0-1.0 
 作用说明：核采样阈值，控制生成内容的多样性和准确性平衡，值越大生成越多样 
 推荐设置：0.85-0.9 
 效果对比：
  - 低 top_p（<0.8）：生成更保守，准确性高
  - 高 top_p（>0.9）：生成更开放，多样性强

参数名：最小采样概率 
 调节范围：0.0-1.0 
 作用说明：最小采样概率，避免低概率词被完全忽略，增加生成的丰富性，默认值通常已足够 
 推荐设置：0.05 
 适用场景：需要增加生成多样性时使用

参数名：典型采样阈值 
 调节范围：0.0-1.0 
 作用说明：典型性采样参数，控制生成内容的典型性，值为1.0时不使用此特性，大多数用户无需调整 
 推荐设置：1.0 

参数名：生成温度 
 调节范围：0.0-2.0 
 作用说明：生成温度，控制生成的随机性，是最常用的参数之一，越高越随机，越低越确定 
 推荐设置：0.6-0.8 
 温度指南：
  - 低温度（0.1-0.4）：适合需要准确答案的场景，如问答、代码生成
  - 中温度（0.5-0.8）：适合大多数场景，平衡准确性和创造性
  - 高温度（0.9-1.5）：适合创造性任务，如故事生成、诗歌创作

参数名：重复惩罚 
 调节范围：0.0-10.0 
 作用说明：重复惩罚，避免生成重复内容，提高文本质量，是常用参数之一 
 推荐设置：1.0 
 调整建议：
  - 当出现重复内容时，逐步增加到1.1-1.3
  - 不宜过高，否则可能导致生成内容不连贯

参数名：频率惩罚 
 调节范围：0.0-1.0 
 作用说明：频率惩罚，减少高频词出现的概率，默认值0.0表示不使用此功能，通常不需要主动调整 
 推荐设置：0.0 

参数名：存在惩罚 
 调节范围：0.0-2.0 
 作用说明：出现惩罚，鼓励生成新内容，增加文本丰富度，是常用参数之一 
 推荐设置：1.0 
 使用场景：
  - 长文本生成时避免主题漂移
  - 对话中鼓励提出新观点

参数名：Mirostat模式 
 调节范围：0-2 
 作用说明：Mirostat采样模式，用于控制生成文本的困惑度，0=关闭，1=基本模式，2=版本2，大多数用户不需要使用 
 推荐设置：0 
 高级用户建议：尝试模式1或2以获得更一致的生成质量

参数名：Mirostat学习率 
 调节范围：0.0-1.0 
 作用说明：Mirostat学习率，仅在mirostat_mode启用时有效，控制调整速度 
 推荐设置：0.1 

参数名：Mirostat目标困惑度 
 调节范围：0.0-10.0 
 作用说明：Mirostat目标困惑度，仅在mirostat_mode启用时有效，控制生成的多样性 
 推荐设置：5.0 

参数名：对话状态ID 
 调节范围：-1-999999 
 作用说明：对话状态ID，用于保存和恢复对话会话状态，仅在需要保持连续对话上下文时有用 
 推荐设置：-1
 会话管理：
  - 使用不同的state_uid可以同时维护多个独立会话
  - 连续对话时保持相同的state_uid以维持上下文

### 参数调节技巧

1. **控制输出长度**：调整 `max_tokens`，值越大输出越长，但占用更多资源。建议根据实际需求设置，避免设置过大导致性能问题

2. **控制随机性**：`temperature` 是最常用的参数之一，越低越确定（适合需要准确答案的场景），越高越有创意（适合生成故事、诗歌等创造性内容）

3. **控制多样性与准确性平衡**：
   - `top_p` 和 `top_k` 通常配合使用
   - 降低 `top_k` 并提高 `top_p` 可以在保证一定多样性的同时提高准确性
   - 提高 `top_k` 并降低 `top_p` 可以增加生成的多样性

4. **避免重复内容**：
   - 增加 `repeat_penalty` 可以减少重复内容
   - 结合 `presence_penalty` 使用可以鼓励模型生成新内容

5. **参数优先级建议**：
   - 优先调整：`temperature`、`max_tokens`、`top_p`/`top_k`
   - 次要调整：`repeat_penalty`、`presence_penalty`
   - 高级用户：`min_p`、`mirostat` 相关参数

6. **常用场景参数设置**：
   - **事实问答**：temperature=0.1-0.3, top_k=10, top_p=0.7
   - **创意写作**：temperature=0.8-1.2, top_k=50, top_p=0.95
   - **代码生成**：temperature=0.2-0.4, top_k=20, top_p=0.8
   - **对话交互**：temperature=0.6-0.8, top_k=30, top_p=0.9

## 四、常见问题与解决方案

### 1. 显存不足（OOM）

**症状**：推理时提示 "out of memory" 或 "OOM"

**解决方案**：
- 降低 `n_gpu_layers` 值，减少 GPU 负载
- 减少 `n_ctx` 值，降低上下文长度
- 降低 `max_tokens` 值，减少生成文本长度
- 使用更小的模型文件
- 降低图片 `max_size` 值

### 2. 推理速度慢

**解决方案**：
- 增加 `n_gpu_layers` 值（如果有足够显存）
- 减少 `n_ctx` 值
- 使用更小的模型
- 关闭不必要的应用程序，释放系统资源

### 3. 生成结果质量差

**症状**：生成的文本质量低、重复、不相关或不符合预期

**解决方案**：

1. **优化提示词**：好的提示词比参数调节更重要，确保提示词清晰、具体，提供足够的上下文信息

2. **调整生成参数**：
   - 如果生成内容过于随机或不相关：降低 `temperature`，减小 `top_p`/`top_k`
   - 如果生成内容过于重复：增加 `repeat_penalty`，提高 `presence_penalty`
   - 如果生成内容不够丰富：提高 `temperature`，增加 `top_p`/`top_k`，考虑调整 `min_p`

3. **增加输出长度**：如果生成内容被截断，增加 `max_tokens` 值

4. **使用更高质量的模型**：尝试使用更大或更专业的模型

5. **检查模型适配性**：确保选择的 `chat_handler` 与使用的模型类型匹配

### 4. 处理图片时出错

**解决方案**：
- 确保已启用 `mmproj` 并选择了正确的视觉编码模型
- 确保 `chat_handler` 与模型匹配
- 检查图片尺寸是否过大，降低 `max_size` 值

## 五、使用建议

### 1. 基础使用建议

1. **逐步调节**：先使用默认参数，然后根据实际效果逐步调整
2. **模型匹配**：确保 `chat_handler` 与使用的模型类型匹配
3. **资源监控**：使用任务管理器监控内存和显存使用情况
4. **提示词优化**：好的提示词比参数调节更重要，建议详细描述需求
5. **会话管理**：长时间使用时定期清理会话状态，释放资源

### 2. 模型选择最佳实践

1. **根据显存选择模型**：
   - 24GB+：任意模型
   - 16GB：中型模型
   - 12GB：中小型模型
   - 8GB：小型模型
   - 4-6GB：超小型模型

2. **根据任务选择模型**：
   - 复杂视觉理解：Qwen3-VL、GLM-4.6V
   - 快速推理：Moondream2、nanoLLaVA
   - 移动设备：MobileVLM、TinyLLaVA

3. **量化级别选择**：
   - 优先：Q4_K_M（平衡大小和质量）
   - 高质量：Q5_K_M、Q6_K
   - 小尺寸：Q3_K_M、Q2_K

### 3. 性能优化技巧

1. **显存管理**：
   - 多模型切换时使用卸载模型节点
   - 大任务后清理会话状态
   - 避免同时加载多个大型模型

2. **速度优化**：
   - 适当降低 n_ctx 提高推理速度
   - 使用更小的模型处理简单任务
   - 调整 image_max_tokens 减少视觉处理开销

3. **质量优化**：
   - 复杂任务使用更大的模型
   - 提高 temperature 增加创意性
   - 使用详细的提示词模板
